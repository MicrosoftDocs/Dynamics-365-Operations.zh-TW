# <a name="onboard-vendors"></a>加入廠商

[!include [banner](../includes/banner.md)]

---

新廠商可以根據從廠商的代表人處收集的資訊，在 Microsoft Dynamics 365 Supply Chain Management 中加入並註冊為廠商。

該流程包括以下步驟，由各種角色在系統中執行動作。

1. **資料管理 OData** – 實體匯入 - 初始要求是潛在廠商註冊要求。 通常，此要求來自允許匿名存取的客戶託管網站等來源。 廠商可以提供基本資訊進行註冊，例如廠商名稱、理由、組織編號以及連絡人的姓名和電子郵件地址。 要求透過資料管理介面匯入。
2. **潛在廠商註冊要求清單頁面**  - 根據潛在廠商註冊要求中提供的資訊，採購專業人員決定廠商是否應加入。 採購專業人員在 **潛在廠商註冊要求** 清單頁面上查看傳入的要求。
3. **使用者佈建工作流程** - 當採購專業人員驗證傳入要求中的資訊並決定繼續加入流程時，使用者要求工作流程會對新使用者進行設定，並傳送邀請電子郵件以接受連絡人作為 Microsoft Dynamics 365 的驗證使用者。
4. **廠商註冊精靈** - 廠商的連絡人使用新的使用者帳戶登入。 他們完成廠商註冊精靈，提供地址、業務資訊、採購類別和問卷回覆等資訊。
5. **核准工作流程** - 建立包含註冊資訊的廠商要求。 此廠商要求會提交到工作流程，並進行路由以供審查和核准。
6. **建立廠商主資料和使用者角色修改** - 廠商要求獲得核准後，將建立廠商記錄。 廠商連絡人的使用者帳戶會被授予廠商共同作業權限或被停用。

下表顯示此流程中涉及的步驟和角色。

| 角色和「流程」       | 資料管理 OData – 實體匯入 | 潛在廠商註冊要求清單頁面 | 使用者佈建工作流程 | 廠商註冊精靈 | 核准工作流程 | 建立廠商主資料和使用者角色修改 |
|--------------------------|---|---|---|---|---|---|
| 系統                   | 匯入新廠商的要求。 | | | | | 接受廠商要求後，將建立廠商記錄。 |
| 採購專業人員 | | 開始加入流程。 | | | 查看並接受或拒絕廠商要求。 | |
| 管理員            | | | 在 Supply Chain Management 和 Microsoft Azure 中建立使用者。 | | | |
| 廠商連絡人    | | | 傳送電子郵件給連絡人。 | 註冊廠商資訊。 | | |

如需廠商加入流程快速示範，請觀看以下的 YouTube 短片: [在財務和營運中加入新廠商](https://www.youtube.com/watch?v=0KUc3AGaTKk)。

## <a name="importing-the-prospective-vendor-registration-request"></a>匯入潛在廠商註冊要求

潛在廠商註冊要求是 Supply Chain Management 中的一個實體。 您可以設定系統透過此實體匯入資料。 

下表顯示該實體包含且可匯入的資訊。

| 欄位                        | 描述 |
|------------------------------|-------------|
| 廠商名稱                  | 廠商名稱。 |
| 業務理由       | 廠商要求的一個或多個原因。 |
| 組織編號          | 官方已知的註冊編號。 |
| 企業營運             | 廠商所處的企業營運範圍。 |
| 連絡人的名字  | 將受邀註冊廠商資訊之人員的名字。 |
| 連絡人的中間名 | 將受邀註冊廠商資訊之人員的中間名。 |
| 連絡人的姓氏   | 將受邀註冊廠商資訊之人員的姓氏。 |
| 連絡人的電子郵件       | 將用於在 Supply Chain Management 中建立新使用者，並且將在租用戶的 Azure Active Directory (Azure AD) 帳戶中註冊的電子郵件地址。 |
| 提交日期               | 在外部系統中建立要求的日期。 |
| 法律實體                 | 廠商要求成為廠商的法律實體。 此值必須是已在 Supply Chain Management 中註冊的法律實體代碼。 如果透過匯入流程未收到任何值，則應用採購和資源開發參數中的值。 |
| 廠商類型                  | 廠商可以是組織或個人。 廠商類型決定了最終如何建立廠商。 |

匯入潛在廠商註冊要求後，它會顯示在 **潛在廠商註冊要求** 清單頁面。 採購專業人員可以從此清單頁面邀請使用者。 用於佈建使用者的使用者要求會傳送到工作流程。

## <a name="submitting-a-prospective-vendor-user-request"></a>提交潛在廠商使用者要求

潛在廠商使用者要求的目的是設定提交初始要求的人員，以便他們可以使用潛在廠商註冊要求中提供的電子郵件帳戶登入 Supply Chain Management。

潛在廠商使用者要求由使用者要求工作流程處理。 此工作流程透過 Azure AD B2B 共同作業。 其會在 Supply Chain Management 中建立具有適當安全性設定的使用者。

設定的新使用者具有以下資訊安全角色：

- 系統外部使用者
- 潛在廠商 (外部)

新使用者將收到一封由使用者要求工作流程產生的電子郵件。 此電子郵件會邀請使用者登入系統。

有關電子郵件設定和一般工作流程的資訊，請參閱[設定和維護廠商共同作業](set-up-maintain-vendor-collaboration.md)中對使用者要求工作流程的描述。

## <a name="vendor-registration"></a>廠商註冊

登入 Supply Chain Management 的潛在廠商使用者將看到廠商註冊精靈的首頁，他們可以在其中輸入廠商資訊。

該精靈反映了廠商要求的設定。 廠商開展業務的國家或地區決定了精靈中要求的資訊以及必填資訊。

有關廠商要求設定的更多資訊，請參閱[設定和維護廠商共同作業](set-up-maintain-vendor-collaboration.md)。 下表概述精靈中的頁面以及各頁面的用途。

| 頁面                       | 描述 |
|----------------------------|-------------|
| 國家/地區             | 國家或地區決定在其餘精靈頁面上套用的廠商要求設定。 它還決定 **稅收狀態** 查詢中的值。 |
| 條款和條件       | 此頁面是否可使用得取決於廠商要求設定。 如果可用，使用者必須確認條款和條件後才能繼續。 |
| 廠商資訊         | 此頁面包含廠商名稱，該名稱是從原始潛在廠商註冊要求中自動輸入的。 它還包含組織編號、廠商的電話號碼、傳真號碼和電子郵件地址，以及用於各種目的的廠商地址。 |
| 連絡人資訊 | 此頁面包含連絡人名稱，該名稱是從原始潛在廠商註冊要求中自動輸入的。 它還包含連絡人的電話號碼和電子郵件地址，以及用於各種目的的連絡人地址。 |
| 業務資訊       | 此頁面包含稅務登記號碼 (適用於各個國家或地區) 和員工人數。 它還表明該企業是否為少數股東所有。 |
| 採購類別     | 此頁面包含廠商要求核准的採購類別。 使用者可以在採購類別階層中選擇類別。 可在 **採購和資源開發** &gt; **設定** 下的 **採購和資源開發參數** &gt; **廠商共同作業**，設定階層中顯示的層級數。 |
| 問卷             | 該精靈可能包括一組針對廠商的問卷。 精靈中顯示的問卷是根據廠商要求或按採購類別設定的。 如果按採購類別設定問卷，則廠商要求核准的採購類別決定精靈中顯示的問卷。 在 **採購類別** 頁面，您可以在相關類別下新增問卷，並將活動類型設定為 **廠商加入**。 |

當潛在廠商使用者完成廠商註冊精靈時，將建立廠商要求。

## <a name="manually-or-automatically-submit-a-vendor-request"></a>手動或自動提交廠商要求

可以將廠商要求建立為草稿並手動提交到工作流程。 或者，在完成廠商註冊精靈後，將廠商要求自動提交到工作流程。 例如，如果採購專業人員想要在將要求提交到工作流程之前，評估是否應透過核准流程傳送要求，則可以手動提交要求。

- 選擇 **採購和資源開發參數** &gt; **廠商共同作業**，然後選擇 **自動將潛在廠商註冊提交到工作流程**，將廠商要求設定為在完成廠商註冊精靈後，將其自動提交到工作流程。

## <a name="vendor-requests"></a>廠商要求

廠商要求可在 **廠商共同作業使用者要求** 頁面使用。

廠商要求包含潛在廠商使用者在廠商註冊精靈中輸入的資訊。

該要求可讓您查看廠商資訊並決定廠商是否應成為註冊廠商。

廠商要求應提交到工作流程，並應發送給相關的審閱者和核准者。 有關如何設定工作流程的基本資訊，請參閱[採購和資源開發工作流程](procurement-sourcing-workflows.md)。

下表顯示廠商要求可能具有的狀態。

| 狀態                     | 描述 |
|----------------------------|-------------|
| 草稿                      | 廠商要求尚未提交。 |
| 要求已提交          | 廠商要求已提交，並正在處理工作流程的第一步。 |
| 待審閱             | 如果一個工作流程工作中有多個審閱者，審閱者可以接受審閱廠商要求的工作，然後完成審閱。 如果只有一位審閱者，該參與者可以選擇工作流程動作中的 **已完成**。 他們不必先接受工作項目。 |
| 要求待核准   | 廠商要求已發送給參與者以供核准，並且具有要求附加資訊的選項。 要求附加資訊會導致工作項目發回至提交者。 處於此狀態時，也可以核准或拒絕廠商要求。 |
| 申請變更要求 | 核准者已要求提供其他資訊，並且廠商要求已發送給提交廠商要求的人。 提交者可以新增所需資訊，然後重新提交廠商要求。 如果重新提交廠商要求，則狀態將改回 **要求待核准** 狀態。 |
| 要求已核准           | 此狀態是最終狀態。 |
| 要求已拒絕           | 此狀態是最終狀態。 |

## <a name="approving-a-vendor-request"></a>核准廠商要求

核准廠商要求後將建立廠商帳戶，並且在初始潛在廠商註冊要求和廠商要求中均顯示 **已核准** 狀態。

核准廠商要求之前，在 **新廠商** 頁面的 **一般** FastTab 上，選擇 **廠商群組** 以選擇廠商群組。

如果潛在廠商使用者應有權作為代表廠商的廠商共同作業使用者存取 Supply Chain Management，請將廠商共同作業存取權限設定為 **是**。 若要停用潛在廠商註冊時使用的使用者帳戶，請將此權限設定為 **否**。

如果廠商共同作業存取權限設定為 **是**，核准廠商要求後，將提交修改使用者角色的要求，以便使用者具有在 **外部角色** 中為 **廠商** 定義的角色。 如果此權限設定為 **否**，核准廠商要求後，將提交要求以停用使用者。 在這種情況下，必須設定工作流程以停用使用者要求。

廠商申請核准後建立廠商帳戶，從廠商要求建立廠商的編號規則必須設定為 **自動**。

有關廠商共同作業使用者的存取權限的概述，請參閱[設定和維護廠商共同作業](set-up-maintain-vendor-collaboration.md)。

## <a name="rejecting-a-vendor-request"></a>拒絕廠商要求

如果拒絕廠商要求，則必須在廠商要求中選擇拒絕原因。

拒絕廠商要求時，將提交要求以停用使用者。 在這種情況下，必須設定工作流程以停用使用者要求。 如需詳細資訊，請參閱[設定和維護供應商共同作業](set-up-maintain-vendor-collaboration.md)。

拒絕廠商要求後，在初始潛在廠商註冊要求和廠商要求中均顯示 **已拒絕** 狀態。

## <a name="deleting-a-prospective-vendor-registration-request-in-various-statuses"></a>刪除處於各種狀態的潛在廠商註冊要求

使用潛在廠商註冊要求上的 **刪除** 動作可以清理和移除已建立的記錄鏈，並且可以停用使用者帳戶。 **刪除** 動作的結果會根據潛在廠商註冊要求的狀態而有所不同，如下表所示。

|      已要求使用者      | 選擇<strong>邀請使用者</strong>後，狀態變更為<strong>使用者要求</strong>，建立潛在使用者要求並將其提交到使用者要求工作流程。 |                                                                                                          您無法刪除具有此狀態的潛在廠商註冊要求，因為使用者要求工作流程尚未結束。                                                                                                          |
